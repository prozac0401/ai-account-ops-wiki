## 개요

AI 계정 운영의 원칙은 단순하다. **필요한 사람에게만, 필요한 기간 동안만, 필요한 권한만 준다.**

말은 쉽지만 실제로는 결제, 초대, 교육 일정, 개인정보 동의, 관리자 권한이 뒤엉킨다. 그래서 처음부터 원칙을 문서로 정해 두는 편이 낫다.

## 1인 1계정 원칙

공용 계정은 되도록 쓰지 않는다.

공용 계정은 편해 보이지만 다음 문제가 생긴다.

- 누가 어떤 프롬프트를 입력했는지 알기 어렵다.
- 사고가 났을 때 책임 추적이 어렵다.
- 서비스 약관상 좌석 공유 문제가 생길 수 있다.
- 교육 종료 후 회수 범위가 불명확해진다.

실습장 운영 때문에 어쩔 수 없이 좌석 계정이 필요하다면, `seat01`, `seat02`처럼 계정을 만들고 배정표를 남긴다. 단, 이 방식은 예외로 보고 별도 승인 기록을 둔다.

## 유료 라이선스와 서비스 접근은 다르다

많이 헷갈리는 부분이다.

- 라이선스를 샀다고 서비스가 바로 켜지는 것은 아니다.
- 서비스를 켰다고 Pro 기능이 바로 생기는 것도 아니다.
- 사용자가 초대를 수락하지 않으면 결제를 해도 실제 사용이 안 될 수 있다.

예를 들어 Colab은 조직에서 Colab 서비스를 켜고, Pro 또는 Pro+ 라이선스를 사용자에게 할당해야 한다. 둘 중 하나만 해서는 부족하다.

## 먼저 1명으로 테스트한다

대량 초대 전에 항상 테스트 계정을 1개 만든다.

추천 순서는 다음과 같다.

1. 테스트 이메일을 수집 목록에 넣는다.
2. 초대 또는 라이선스 할당을 한다.
3. 사용자가 실제로 로그인한다.
4. Pro 기능이나 팀 워크스페이스가 보이는지 확인한다.
5. 회수한다.
6. 회수 후 사용 불가 상태가 되었는지 확인한다.
7. 그 다음 전체 대상자에게 적용한다.

## 결제 권한과 운영 권한을 나눈다

가능하면 결제 담당자와 교육 운영자를 분리한다.

| 역할 | 맡는 일 |
|---|---|
| 결제 담당자 | 카드 등록, 결제 실패 대응, 인보이스 확인, 선결제 판단 |
| 서비스 관리자 | 좌석 구매, 멤버 초대, 라이선스 할당, 권한 회수 |
| 교육 담당자 | 수강생 명단 수집, 안내, 동의서 징구, 현장 문의 대응 |
| 보안/법무 | 개인정보, 국외 이전, 약관, 사고 대응 검토 |

사람이 적으면 한 명이 여러 역할을 할 수 있다. 그래도 문서상 역할은 나눠 두는 것이 좋다. 그래야 사고가 났을 때 “누가 무엇을 확인했는지”가 남는다.

## 개인 Gmail 사용 시 주의

수강생이 개인 Gmail을 쓰면 운영은 쉬워진다. 하지만 통제는 약해진다.

- 조직 SSO가 적용되지 않을 수 있다.
- 조직 DLP가 적용되지 않을 수 있다.
- 계정 자체의 2단계 인증을 강제하기 어렵다.
- 교육 종료 후 계정 자체 삭제를 관리자가 대신할 수 없다.

그래서 개인 Gmail을 쓸 때는 최소한 아래를 챙긴다.

- Google Form 수집 시 개인정보 고지
- 수강생 본인의 2단계 인증 권장 또는 의무 안내
- 실제 업무정보·개인정보·비밀키 입력 금지
- 교육 종료 후 서비스 멤버 제거
- 회수 완료 기록 보관

## 공식 문서와 실제 사례를 구분한다

예를 들어 “Colab Pro를 처음에는 2개밖에 못 샀다”는 프로젝트 실제 사례다. 공식 문서상 일반 규칙이라고 단정하면 안 된다.

반대로 “Colab 온라인 직접 구매는 Annual/Fixed-Term Plan을 선택해야 한다”는 공식 문서에 있는 내용이다.

이 위키에서는 다음 표현을 쓴다.

| 표현 | 의미 |
|---|---|
| 공식 기준 | 공식 문서에 명시되어 있음 |
| 프로젝트 실제 사례 | 이 프로젝트에서 실제로 관찰됨 |
| 운영 판단 | 공식 기준과 실제 상황을 바탕으로 한 내부 권장 |
| 확인 필요 | 서비스 정책이나 계약 상태에 따라 달라질 수 있음 |

## 금지 데이터 원칙

수강생에게 가장 먼저 말해야 하는 것은 기능이 아니다. “무엇을 넣으면 안 되는가”다.

입력 금지 예시는 다음과 같다.

- 주민등록번호, 여권번호, 전화번호 같은 개인정보
- 고객명, 고객 연락처, 상담 이력
- 회사 내부 문서 원문
- API 키, 토큰, 비밀번호, 인증서
- 비공개 소스코드 전체
- 계약서, 제안서, 견적서의 민감 내용

실습에는 별도 샘플 데이터를 쓴다. 실제 업무 데이터를 쓰면 교육이 아니라 사고 대응 훈련이 될 수 있다.

## 증빙은 운영의 일부다

운영이 끝난 뒤 남겨야 할 자료는 다음과 같다.

- 수강생 명단
- 동의서 또는 고지 확인 기록
- 초대 일시
- 라이선스 할당 일시
- 회수 일시
- 결제 내역 또는 인보이스
- 공식 정책 링크와 확인 날짜
- 보안 검토 의견
- 사고 또는 예외 처리 기록

구두로 “다 지웠습니다”라고 말하는 것은 운영 증빙이 아니다.

## 자주 헷갈리는 점

- 멤버 제거는 계정 삭제가 아니다.
- 좌석 삭제는 결제 중단과 다를 수 있다.
- API 키 삭제는 프로젝트 삭제와 다르다.
- 서비스 ON은 라이선스 할당과 다르다.
- Privacy Mode는 “전송 없음”이 아니라 “보관·학습 제한”에 가깝다.
- 수강생에게 영어 개인정보처리방침 링크만 던지는 것은 좋은 고지가 아니다.
