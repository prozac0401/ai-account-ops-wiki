## 개요

AI 계정 운영은 혼자서 다 할 수 있어 보인다. 하지만 실제로는 관리자, 교육 담당자, 결제 담당자, 법무, 정보보안, 교육생이 모두 조금씩 얽힌다.

역할을 나누지 않으면 문제가 생겼을 때 모든 질문이 한 사람에게 몰린다. 더 큰 문제는, 누가 승인했고 누가 회수했는지 나중에 설명하기 어렵다는 점이다.

## 핵심 역할

### 계정 관리자

서비스 콘솔에서 직접 계정을 다루는 사람이다.

주요 업무는 다음과 같다.

- 워크스페이스 생성
- 멤버 초대
- 라이선스 할당
- API 프로젝트 생성
- API 키와 서비스 계정 관리
- 관리자 역할 부여
- 교육 종료 후 멤버 제거
- 회수 결과 기록

계정 관리자는 권한이 세다. 그래서 “편의상 교육 담당자에게 Owner 권한을 다 주는 방식”은 권장하지 않는다.

### 결제 담당자

돈이 나가는 경로를 관리한다.

주요 업무는 다음과 같다.

- 카드 등록
- 자동 결제 실패 대응
- 선결제 또는 수동 결제
- 인보이스 확인
- 좌석 추가 시 비용 확인
- 연간계약과 월간계약 차이 설명
- 환불·해지 가능 여부 확인

AI SaaS 결제는 생각보다 자주 막힌다. 카드 인증, 해외 결제, 신규 계정 제한, 수량 제한, 연간계약 조건이 한꺼번에 나올 수 있다.

### 교육 담당자

교육생과 가장 가까운 사람이다.

주요 업무는 다음과 같다.

- 교육생 명단 수집
- Google Form 또는 신청서 운영
- 개인정보 고지·동의 안내
- 교육생 사전 안내문 배포
- 접속 테스트 안내
- 현장 로그인 문제 1차 대응
- 교육 종료 후 회수 대상 확인

교육 담당자가 관리자 권한까지 가질 필요는 없다. 다만 교육생 명단과 교육 일정은 교육 담당자가 가장 정확히 안다.

### 정보보안 담당자

서비스 사용의 위험을 보는 사람이다.

주요 업무는 다음과 같다.

- 입력 금지 데이터 기준 확인
- DPA, 보안 인증, 암호화, 로그 정책 검토
- 외부 SaaS 사용 승인
- 사고 대응 연락 체계 확인
- 민감정보 유입 시 대응 방식 검토

교육장이 보안상 비교적 자유로운 환경이어도 확인은 필요하다. 자유로운 환경이라는 말이 사고 대응이 필요 없다는 뜻은 아니다.

### 법무·개인정보 담당자

개인정보 고지와 동의의 형식을 보는 사람이다.

주요 업무는 다음과 같다.

- 개인정보 수집·이용 목적 검토
- 수집 항목 검토
- 보유·이용 기간 검토
- 동의 거부권 및 불이익 문구 검토
- 국외 이전 고지 검토
- 제3자 제공 또는 처리위탁 구조 검토

OpenAI, Cursor, Anthropic 같은 해외 SaaS를 쓰면 국외 이전 문제가 거의 항상 등장한다.

### 교육생

실제로 서비스를 쓰는 사람이다.

교육생의 책임은 어렵게 쓰면 안 된다. 아래 정도로 명확히 말한다.

- 본인 계정만 사용한다.
- 다른 사람에게 계정을 빌려주지 않는다.
- 실습용 샘플 데이터만 입력한다.
- 개인정보, 고객정보, 회사기밀, API 키를 입력하지 않는다.
- 이상한 응답이나 사고 의심 상황은 바로 알린다.
- 교육 종료 후 접근 권한이 회수될 수 있음을 이해한다.

## RACI 표

| 업무 | 계정 관리자 | 결제 담당자 | 교육 담당자 | 보안 | 법무/개인정보 | 교육생 |
|---|---|---|---|---|---|---|
| 서비스 선정 | C | C | R | C | C | - |
| 계약/결제 | C | R | C | C | C | - |
| 개인정보 고지 | C | - | R | C | A | I |
| 이메일 수집 | C | - | R | C | C | I |
| 계정 초대 | R | - | C | - | - | I |
| 라이선스 할당 | R | C | C | - | - | - |
| 교육 안내 | C | - | R | C | C | I |
| 사용 중 사고 대응 | R | C | R | A | C | I/R |
| 교육 종료 회수 | R | C | R | C | - | I |
| 증빙 보관 | R | R | R | C | C | - |

- R: 실행 담당
- A: 최종 책임
- C: 협의
- I: 통보

## 서비스별 관리자 역할 비교

| 서비스 | 강한 관리자 역할 | 일반 사용자 역할 | 주의점 |
|---|---|---|---|
| Google Workspace | Super Admin, Billing Admin | User | Super Admin은 최소 인원만 둔다 |
| ChatGPT Business | Owner, Admin | Member | Owner는 결제와 워크스페이스 설정까지 볼 수 있다 |
| OpenAI API | Organization Owner, Project Owner | Project Member, Reader | API 키 생성 권한을 아무에게나 주지 않는다 |
| Cursor | Team Admin, Enterprise Admin | Member | Enterprise가 아니면 SCIM·감사로그 기능이 제한될 수 있다 |
| Claude Business | Primary Owner, Owner | Member | Primary Owner의 데이터 export 권한을 이해해야 한다 |

## 자주 헷갈리는 점

### 교육 담당자가 Owner여야 하나?

대부분 아니다.

교육 담당자는 명단과 일정 관리가 핵심이다. 계정 콘솔의 Owner 권한은 결제, 사용자, 데이터 설정까지 만질 수 있으므로 최소화하는 편이 좋다.

### 결제 카드 등록자는 곧 관리자라고 봐야 하나?

아니다.

카드 등록과 사용자 관리는 다른 일이다. 다만 작은 조직에서는 같은 사람이 둘 다 맡을 수 있다. 이 경우에도 문서상으로는 두 역할을 나눠 적는다.

### 교육생에게도 책임을 물을 수 있나?

가능한 범위에서 사전에 안내해야 한다. 특히 입력 금지 데이터는 교육 시작 전, 실습 직전, 안내문에 반복해서 알려야 한다.

## 운영 팁

첫 교육에서는 역할을 너무 복잡하게 만들지 않는다. 대신 아래 세 명은 꼭 지정한다.

1. 계정/기술 담당
2. 교육/현장 담당
3. 결제/승인 담당

이 세 명만 명확해도 대부분의 현장 혼선은 줄어든다.
