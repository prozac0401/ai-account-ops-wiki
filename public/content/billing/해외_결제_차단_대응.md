[목차]

## 개요

해외 결제 차단은 교육 당일에 해결하기 어렵다. 카드사, 결제 담당자, 계정 담당자, 과정 담당자가 모두 관여해야 하고, 서비스별로 셀프서비스 카드 결제만 가능한 경우도 있기 때문이다.

이 문서는 결제 화면에서 해외 카드 결제가 막혔을 때 어떤 순서로 확인하고, 어떤 대체 경로를 준비할지 정리한다. 목표는 “결제가 안 된다”에서 멈추지 않고, 원인을 분류해 교육 일정에 영향을 덜 주는 것이다.

## 먼저 구분할 것

결제 실패 원인은 하나가 아닐 수 있다. 같은 “카드 승인 실패”라도 아래 항목을 나눠 확인한다.

| 구분 | 의심할 지점 | 확인할 곳 |
|---|---|---|
| 해외 온라인 결제 차단 | 법인카드가 해외 온라인 거래를 막고 있음 | 카드사 앱, 카드사 콜센터, 법인카드 관리자 |
| 정기결제 차단 | 구독형 SaaS(Software as a Service)<sup class="wiki-footnote-ref"><a class="wiki-footnote-link" id="abbr-ref-4" href="#abbr-note-4">[4]</a></sup> 자동 청구가 막힘 | 카드사 반복 결제 허용 설정 |
| 한도 초과 | 1회, 1일, 월간 한도 또는 잔여 한도 부족 | 카드 한도, 승인 가능 금액 |
| 본인 인증 실패 | 3D Secure, 앱카드, OTP(One-Time Password)<sup class="wiki-footnote-ref"><a class="wiki-footnote-link" id="abbr-ref-1" href="#abbr-note-1">[1]</a></sup>, 문자 인증 실패 | 카드 인증 수단, 결제 담당자 휴대폰 |
| 청구 정보 불일치 | 청구 주소, 국가, 우편번호, 카드 소유자 정보 불일치 | 서비스 Billing 화면, 카드사 등록 정보 |
| 결제 수단 제한 | 셀프서비스 플랜이 카드만 받음 | 서비스 공식 Billing 문서 |
| 위험 거래 탐지 | 짧은 시간에 여러 계정·좌석을 결제해 이상 거래로 보임 | 서비스 지원, 카드사 승인 로그 |

## 교육 전 예방 전략

해외 결제는 교육 최소 7영업일 전에 한 번 끝까지 테스트한다. 단가 확인만 하지 말고 실제로 소량 결제를 통과시켜야 한다.

1. 결제 담당자와 카드사 연락 가능 시간을 확보한다.
2. 법인카드의 해외 온라인 결제, 정기결제, 소프트웨어/SaaS(Software as a Service) 업종 제한을 확인한다.
3. 1회 한도, 1일 한도, 월간 한도, 남은 한도를 확인한다.
4. 결제 인증을 받을 휴대폰, 카드사 앱, OTP(One-Time Password), 문자 수신 가능 여부를 확인한다.
5. 계정 결제 담당자는 실물 카드를 소지한다. 카드사나 서비스 결제 화면에서 카드 실소유 여부 확인을 위해 카드번호 전체를 요구할 수 있다.
6. 카드번호 전체는 공식 카드사 채널 또는 공식 서비스 Billing 화면에 결제 담당자가 직접 입력한다. 이메일, 메신저, 문서, 스크린샷으로 공유하지 않는다.
7. 결제 화면의 서비스명, 좌석 수, 금액, 통화, 세금 표시를 캡처한다.
8. 수강생 전체 수량으로 바로 결제하지 않고 운영자 1명과 예비 1~2명 수준으로 테스트한다.
9. 인보이스나 영업 계약이 필요한 서비스는 교육 직전에 시작하지 않는다.
10. 결제 실패 시 쓸 예비 카드, 예비 서비스, 무료 구간 운영안을 정한다.

## 서비스별 결제 전략

| 서비스 | 결제 차단 시 보는 관점 | 준비할 대체 경로 |
|---|---|---|
| Google Workspace / Colab | Google Admin console의 결제 계정, 기본 결제 수단, 결제 정보 검증, 수동 결제 가능 여부를 본다. Google은 해외 거래, 온라인 거래, 자동 결제 차단을 결제 실패 원인으로 안내한다. | 백업 카드 추가, 카드사 해외·자동 결제 허용, 결제 정보 검증 완료, 필요한 경우 Google 지원 문의 |
| ChatGPT Business | 셀프서비스 Business는 현재 신용/체크카드 결제만 지원한다. 인보이스, 은행 송금, wire, ACH(Automated Clearing House)<sup class="wiki-footnote-ref"><a class="wiki-footnote-link" id="abbr-ref-2" href="#abbr-note-2">[2]</a></sup>, 구매 주문서, net terms, 분할 결제는 셀프서비스 Business에서 제공되지 않는다. | 카드 결제가 불가하면 ChatGPT Enterprise 또는 ChatGPT Education 영업 상담을 검토 |
| OpenAI API Platform | 사용량 기반이므로 카드 등록, 선결제 크레딧, 예산·한도 설정을 함께 본다. 결제 수단이 막히면 API 실습 전체가 중단될 수 있다. | 사전 크레딧 충전, 예산 제한, 실습용 키 분리, 결제 담당자 대기 |
| Cursor Teams | Teams는 사용자당 월 과금이며 셀프서비스 결제는 카드 중심으로 운영한다. Cursor 가격표에서는 Enterprise에 Invoice/PO(Purchase Order)<sup class="wiki-footnote-ref"><a class="wiki-footnote-link" id="abbr-ref-3" href="#abbr-note-3">[3]</a></sup> billing을 별도 기능으로 안내한다. | 카드 결제 통과가 어렵다면 Enterprise 문의, 좌석 수 축소, 무료 구간 실습안 준비 |
| Claude Team / API | Claude 유료 구독은 Settings > Billing에서 결제 수단과 청구서를 확인한다. Pro/Max는 신용/체크카드 결제만 받는다고 안내되어 있고, API는 Stripe 청구서를 통해 결제한다. | 팀 구독과 API 결제 계정을 분리해 확인, 결제 실패 시 Billing 상태와 청구서 확인 |

## 막혔을 때 10분 안에 할 일

교육 전 결제 테스트나 교육 중 응급 상황에서 결제가 막히면, 무작정 여러 번 다시 누르지 않는다. 같은 카드로 반복 실패하면 카드사나 서비스 위험 탐지에 걸릴 수 있다.

1. 오류 화면을 캡처한다.
2. 서비스명, 계정, 구매 좌석 수, 금액, 통화, 시각, 카드 끝 4자리를 기록한다.
3. 카드사에 해외 온라인 결제, 정기결제, 3D Secure, 한도, 의심 거래 차단 여부를 확인한다.
4. 청구 주소, 국가, 우편번호, 카드 소유자명, 세금 정보가 카드사 등록 정보와 맞는지 확인한다.
5. 서비스 Billing 화면에서 미결제 청구서, 결제 실패 알림, 기본 결제 수단 상태를 확인한다.
6. 카드사 차단 해제가 확인되기 전에는 같은 카드로 반복 시도하지 않는다.
7. 지원 문의가 필요하면 캡처, 시각, 금액, 오류 문구, 카드 끝 4자리만 전달한다. 전체 카드 번호와 인증번호는 공유하지 않는다.

## 카드사에 전달할 문장

카드사에는 기술 서비스명이 낯설 수 있으므로 아래처럼 짧게 설명한다.

```text
해외 온라인 SaaS(Software as a Service) 구독 결제가 차단되었습니다.
결제처는 [서비스명]이고, 결제 통화는 USD입니다.
법인카드 해외 온라인 결제, 반복 결제, 3D Secure, 1회/1일 한도, 의심 거래 차단 여부를 확인해 주세요.
결제 시각은 [한국시간 YYYY-MM-DD HH:MM], 금액은 [금액], 카드 끝 4자리는 [1234]입니다.
```

카드사 상담 후에는 “차단 해제 완료”, “한도 증액 완료”, “온라인 결제 허용 완료”처럼 조치 결과를 문서에 남긴다.

## 회계·구매팀에 미리 공유할 내용

일부 AI 서비스는 빠른 카드 결제를 전제로 만든 셀프서비스 플랜이다. 내부 구매 절차가 인보이스, 구매 주문서, 선지급 승인, 세금계산서 중심이라면 일정이 맞지 않을 수 있다.

| 확인 항목 | 왜 필요한가 |
|---|---|
| 카드 결제 가능 여부 | ChatGPT Business 같은 셀프서비스 플랜은 카드만 받는 경우가 있다. |
| 인보이스 필요 여부 | 인보이스가 필요하면 Enterprise, Education, 영업 계약이 필요할 수 있다. |
| 승인 금액 | 좌석 수, 세금, 환율, 카드 수수료, 예비 좌석을 포함해야 한다. |
| 결제 주기 | 월간, 연간, 선결제, 사용량 과금이 섞일 수 있다. |
| 환불 가능성 | 대부분의 좌석 구독은 사용하지 않은 좌석도 환불되지 않을 수 있다. |
| 증빙 자료 | 결제 화면, 영수증, 청구서, 공식 가격 링크를 함께 보관한다. |

## 교육 중 응급 대응

교육 중에는 결제 문제 자체를 해결하기보다 수업을 멈추지 않는 대응이 우선이다.

1. 유료 기능이 막힌 수강생을 예비 계정이나 예비 좌석으로 옮긴다.
2. 유료 모델이 꼭 필요하지 않은 실습은 무료 구간 또는 다른 서비스로 전환한다.
3. API 실습은 예산이 남아 있는 별도 키로 제한 범위 안에서 진행한다.
4. 운영자 결제 계정, 법인카드 정보, OTP(One-Time Password) 화면을 수강생에게 노출하지 않는다.
5. 결제 담당자가 없는 시간에는 신규 결제를 시도하지 않고, 실습 범위를 줄인다.
6. 영향을 받은 계정, 좌석, 실습 단계를 기록해 교육 후 정산과 재발 방지에 사용한다.

## 재시도 기준

재시도는 “원인을 하나 고친 뒤 한 번”이 원칙이다.

- 카드사에서 해외 온라인 결제 또는 정기결제 차단을 해제했다.
- 1회/1일 한도 문제가 해결됐다.
- 청구 주소와 세금 정보가 수정됐다.
- 서비스 Billing 화면의 미결제 청구서 또는 결제 실패 상태를 확인했다.
- 브라우저, 네트워크, 카드, 계정을 동시에 바꾸지 않고 하나씩 바꿨다.

재시도 후에도 실패하면 새로운 카드를 계속 넣기보다 서비스 지원이나 영업 경로를 검토한다.

