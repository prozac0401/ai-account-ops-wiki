# 위키 문서 작성 스타일 지침

이 문서는 AI 계정 운영 위키의 문체, 구조, 링크, 각주, 검증 기준을 정리합니다. 목표는 교육 운영자, 계정 담당자, 결제 담당자가 빠르게 판단하고 실행할 수 있는 문서를 유지하는 것입니다.

## 기본 방향

- 운영자가 바로 쓸 수 있게 씁니다. 배경 설명보다 확인 순서, 판단 기준, 기록 양식을 우선합니다.
- 공식 기준과 현장 경험을 구분합니다. 공식 문서로 확인된 내용은 출처를 달고, 경험으로 파악한 내용은 “현장 경험상”, “공개 문서로 확인된 보편 규칙은 아님”처럼 범위를 밝힙니다.
- 금액, 정책, 지원 범위처럼 바뀔 수 있는 내용은 확인 기준일을 적습니다.
- 수강자를 가리킬 때는 `학생` 대신 `수강생`을 씁니다.
- 저장소나 문서 묶음을 가리킬 때는 `이 위키`, `이 저장소`, `운영 문서`처럼 씁니다.
- 모호한 표현은 줄입니다. 예를 들어 `비용 감각`보다 `비용 규모`, `Period`보다 `과금 주기`처럼 씁니다.
- 개요성 문서는 `전체 지도`보다 `운영 개요`, `과금 운영 개요`처럼 용도가 드러나는 제목을 씁니다.

## 문서 구조

위키에 노출되는 Markdown 문서는 보통 아래 순서를 따릅니다.

```md
# 문서 제목

> 상위 문서: [[../Home|AI 계정 운영]]
> 분류: 큰 분류 / 작은 분류

[목차]

## 개요

## 사전 준비

## 운영 중 확인

## 종료 후 확인

## 자주 헷갈리는 점

## 약어 각주

## 참고 출처
```

모든 문서가 같은 제목을 가져야 하는 것은 아닙니다. 다만 `개요`, 실행 순서, 예외 상황, 관련 문서, 출처가 한눈에 보이도록 둡니다.

## 제목과 번호

- 화면에서 본문 제목과 오른쪽 목차에 번호가 자동으로 붙습니다.
- 제목 앞에 `1.`, `2.` 같은 수동 번호를 붙이지 않습니다.
- 실제 절차는 제목이 아니라 본문 목록에서 번호를 씁니다.
- 제목은 짧게 씁니다. `교육 시작 전에 계정 담당자가 확인해야 하는 사항`보다 `교육 전 확인`이 낫습니다.
- H1은 문서 제목에만 쓰고, 본문은 `##`, `###` 중심으로 구성합니다.

## 문체

- 기본 문장은 `한다`, `본다`, `확인한다`, `기록한다`처럼 실행형으로 씁니다.
- 운영 판단이 필요한 문장은 단정과 조건을 분리합니다.
- 불필요한 마케팅 문구나 감탄형 표현을 쓰지 않습니다.
- “무조건”, “항상 가능”, “완전히 안전”처럼 증명하기 어려운 표현은 피합니다.
- 위험이 있는 내용은 조건을 먼저 둡니다. 예를 들어 Unpaid Admin 전환은 시건, 로그아웃, 세션 정리 조건과 함께 설명합니다.

## 용어와 약어

- 약어는 처음 중요한 위치에서 `약어(전체 단어)`로 씁니다.
- 어려운 약어는 본문에 긴 설명을 넣지 말고 문서 하단 `약어 각주`에 설명합니다.
- `AI`, `API`, `PC`, `USD`처럼 문맥상 익숙한 약어는 필요할 때만 각주를 답니다.
- 서비스명과 공식 메뉴명은 가능한 한 실제 화면에 가까운 이름을 씁니다.

## 각주

약어 각주는 나무위키처럼 본문에는 번호 링크만 표시하고, 하단에 설명을 모읍니다.

본문 예시:

```md
DLP(Data Loss Prevention) <sup class="wiki-footnote-ref"><a class="wiki-footnote-link" id="abbr-ref-1" href="#abbr-note-1">[1]</a></sup>
```

하단 예시:

```md
## 약어 각주

<div class="wiki-footnotes">
<p class="wiki-footnote" id="abbr-note-1"><a class="wiki-footnote-link" href="#abbr-ref-1">[1]</a> DLP(Data Loss Prevention): 민감정보가 외부로 나가지 않도록 탐지하거나 차단하는 보안 기능이다.</p>
</div>
```

작성 규칙은 다음과 같습니다.

- 번호는 문서마다 `1`부터 시작합니다.
- 본문 단어와 `<sup class="wiki-footnote-ref">` 사이에는 공백을 1칸만 둡니다.
- 본문의 `abbr-ref-1`, `abbr-note-1`과 하단의 번호를 맞춥니다.
- 각주는 본문에 처음 등장하는 순서대로 붙입니다.
- 같은 약어가 여러 번 나오면 보통 첫 번째 의미 있는 위치에만 각주를 답니다.
- 하단 설명은 표로 만들지 않고 `[번호] 설명` 형태로 씁니다. `[번호]` 뒤에 하이픈은 넣지 않습니다.
- `## 약어 각주`는 보통 `## 참고 출처` 바로 위에 둡니다.

## 링크

- 위키 내부 문서 링크는 `[[문서ID|라벨]]` 형식을 씁니다.
- `문서ID`는 `src/main.js`의 `docs`에 등록된 `id`와 정확히 같아야 합니다.
- 같은 문서 안의 제목으로 이동할 때는 일반 Markdown 앵커 링크를 씁니다.
- 외부 사이트는 일반 Markdown 링크나 참고 출처 URL로 남깁니다.
- 관련 문서는 본문 중간에 흩뿌리기보다 `## 관련 문서`에 모으는 편이 읽기 쉽습니다.

## 표와 기록 양식

- 비교, 체크리스트, 정책 차이는 표로 정리합니다.
- 표가 너무 넓어질 때는 열을 줄이고 설명 문단을 따로 둡니다.
- 현장에서 그대로 복사할 기록 양식은 `csv`, `text` 코드 블록으로 제공합니다.
- 코드 블록은 실제 입력 예시나 기록 양식에만 씁니다.

## 과금·보안·개인정보 문서

- 가격, 좌석 과금, 크레딧, True-up, 환불은 공식 문서와 현장 경험을 나눠 씁니다.
- “보편 규칙”이라고 단정하려면 공식 근거가 있어야 합니다.
- 결제 흐름은 월간, 연간, 과금 주기, 일할 계산, 크레딧 차감을 구분합니다.
- 개인정보나 보안 관련 문서는 담당자, 시점, 기록 항목을 함께 씁니다.
- 수강생이 실수할 수 있는 행동은 금지 목록이 아니라 예방 안내와 응급 대응까지 함께 적습니다.

## 검증 체크리스트

문서를 올리기 전에는 아래를 확인합니다.

- `npm run build`가 성공한다.
- 새 문서가 `src/main.js`의 `docs`와 `sections`에 등록되어 있다.
- 제목에 수동 번호가 중복으로 들어가지 않았다.
- 내부 링크의 `id`가 실제 등록값과 같다.
- 약어 각주는 표가 아니라 `[번호] 설명` 형태다.
- 공식 정책이나 가격을 바꿨다면 참고 출처를 확인했다.
- 배포 후 GitHub `Actions`가 성공했는지 확인했다.
